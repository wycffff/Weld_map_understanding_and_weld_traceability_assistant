# 项目回顾

## 1. 起点

这个项目一开始并不是“接手一个现成代码库”，而是从客户需求文档和一组零散样本开始的。

起步阶段最大的不确定性有几个：

- 工业图纸没有统一标准
- 真实样本少，而且格式差异很大
- OCR 在不同图纸上的表现波动很大
- 当前本地小模型很慢、很弱
- 最终目标运行环境会比当前开发机更强

所以一开始就没有把它当成“做一个万能模型”来推进，而是把它当成一个模块化工程系统来建设。

## 2. 最早确立的原则

最重要的第一条原则是：

- `OCR 主，VLM 辅`

这条原则几乎决定了后面所有架构选择。

原因很直接：

- 焊缝编号、图号、BOM tag、数量、材料这些字段都要求精确转录
- 小型本地 VLM 可以帮助判断图纸类型、解释 review、做有限补全，但不适合作为精确字段的唯一来源
- 不确定结果必须进入 `needs_review`，而不是被悄悄硬判

第二条原则是通过显式契约解耦模块：

- `InputDocument`
- `PreprocessedDocument`
- `LayoutPlan`
- `OCRResult`
- `VLMResult`
- `StructuredDrawing`

这让整个系统可以一边演进一边替换局部能力，而不会牵一发动全身。

## 3. 项目是怎么一步步建起来的

### 阶段 A：先搭出可运行的解析骨架

最初先把主链路跑通：

- `M1` 输入与归档
- `M2` 预处理
- `M3` ROI 规划
- `M4` OCR 提取
- `M6` 融合

这个阶段的目标不是做漂亮，而是做到“不依赖数据库、不依赖 VLM，也能稳定输出结构化结果”。

### 阶段 B：从解析器变成应用

接着补上：

- `M7` SQLite 持久化
- `M9` 导出
- `M10` Streamlit UI

从这一步开始，系统不再只是“能跑 JSON”，而是能：

- 上传图纸
- 解析
- 入库
- 搜索
- 导出

### 阶段 C：把 review 和人工接管做成正式流程

真实样本一进来，很快就发现“识别不全”不是偶发，而是常态。

所以后面很重要的一步不是继续盲目追求自动化，而是把这些流程做成产品能力：

- 手动登记焊缝
- 批量补录缺失焊缝
- review queue 的 resolve / reopen
- append-only 事件日志
- 照片证据绑定
- 焊缝状态和检验状态更新

从这里开始，系统具备了业务闭环，而不是只会展示解析结果。

### 阶段 D：先分类，再走不同解析路径

随着图纸种类增加，必须在进入细粒度 ROI 之前就先判断“这到底是什么图”。

于是增加了分类层，基于 OCR 关键词做路由，支持：

- `simple_spool`
- `fabrication_weld_map`
- `pipeline_isometric`
- `dual_isometric`
- `weld_log`

这一步的重要意义在于：

- 不同图纸走不同策略
- 不支持的图可以被明确拒绝
- 失败路径不再是“静默空结果”，而是“明确告警 + 人工接管”

### 阶段 E：把 M5 从接口变成受限能力

`M5` 一开始只是架构位，后面才真正接入主流程。

但接入方式不是让它自由发挥，而是严格受限：

- title block fallback
- weld list 辅助
- weld 位置描述
- review item 解释

同时加了：

- 硬超时
- 输出长度限制
- 按任务启用

这样做的原因很现实：当前本地 Ollama 小模型是 CPU-bound，如果不约束，整个 UI 会被拖死。

### 阶段 F：把“感觉更好了”变成“可以量化评估”

项目后期最关键的一步，是建立了可复现评估：

- 本地 ground truth
- `evaluate-samples`
- 单样本报告
- 聚合指标

从这一步开始，我们不再靠主观印象判断进展，而是能明确看到：

- 哪一步变好了
- 哪一步退步了
- 新样本到底暴露了什么缺口

## 4. 今天已经建成的几条管道

### 解析管道

```text
输入文件
  -> ingestion
  -> preprocessing
  -> preview OCR
  -> 图纸分类
  -> ROI 规划 / 路由
  -> OCR 提取
  -> 可选 bounded VLM
  -> 融合 + 归一化 + review 项
  -> 结构化结果
```

### 追溯业务管道

```text
结构化结果
  -> SQLite 持久化
  -> UI 搜索 / 浏览
  -> 缺失焊缝时手动补录
  -> 状态更新
  -> 检验更新
  -> 照片证据绑定
  -> append-only 事件日志
  -> 导出
```

### 评估管道

```text
样本图纸
  -> 解析
  -> 与 ground truth 对比
  -> 单样本报告
  -> 聚合指标
```

## 5. 几张关键样本分别教会了我们什么

### 1.jpg

这张图帮助我们建立了第一条最小闭环。

它让我们确认：

- 简单图纸上 OCR-first 是可行的
- 页面上的每个编号都不能直接当 weld
- ground truth 必须写清楚，否则阀门 tag 也会被误当焊缝

### 2.jpeg / 6.jpeg

这类 fabrication 图驱动了：

- BOM 列语义映射
- 焊缝框提取
- 重复样本处理

### 3.png

这张图说明了 `pipeline_isometric` 和普通 weld-box 图不是同一类问题。

它推动了：

- 单独的图纸分类
- welding list 路径
- review-first 的数字焊缝推断

### 4.webp

这张低清晰度双图页让我们认识到：

- 有些输入不应该强行“装作识别成功”
- 明确 fallback 比错误高置信度输出更重要
- 人工接管路径必须是正式能力

### 11.png

这张图是后期最重要的样本之一，因为它暴露出一个之前完全没覆盖到的缺口：

- 字母焊缝编号
- 分组 BOM
- 英制长度数量格式

它的价值很大，因为它让原本“看起来很好看”的指标突然掉下来，从而暴露了真实的结构性问题。

### hjrz.webp

这张 `WELD LOG` 样本的价值非常高：

- 它既是导出模板
- 也是 UI 参考模板

它直接推动了后面的 WELD LOG 导出和 WELD LOG 风格工作台视图。

## 6. 我们是怎么量化每一步验收结果的

评估不是一次性做出来的，而是逐步演化出来的。

核心指标包括：

- 图号准确率
- 图纸类型准确率
- weld precision
- weld recall
- BOM 字段准确率

真正重要的转折不是“数字变高”，而是“数字开始可信”：

- 最早只有少量样本时，结果看起来几乎完美，但覆盖面不够
- 增加 `11.png` 以后，BOM 指标明显下降，暴露了 grouped BOM 的结构问题
- 修完 grouped BOM 之后，指标重新回升，而且这次是更有代表性的真实提升

这也是项目后期最宝贵的一点：我们接受了“加新样本导致指标暂时变差”这件事，因为那说明测试集终于开始接近真实世界。

## 7. 当前量化结果

在当前可用的本地标注样本集上，最新结果是：

- `drawing_number_accuracy = 1.0`
- `drawing_type_accuracy = 1.0`
- `weld_precision_micro = 1.0`
- `weld_recall_micro = 1.0`
- `bom_field_accuracy_micro = 0.8154`

这意味着：

- 主要验收目标在当前样本集上已经达成或超过
- 系统已经不是“想法验证”
- 而是一个具备可量化质量的本地追溯工作流

## 8. 为什么现在是一个合适的暂停点

目前是一个很自然的阶段性收尾点，因为：

- `M1-M10` 都已经落地并能运行
- 当前可用 benchmark 已经达到主要目标
- UI、数据库、导出、review、照片、状态更新都已经闭环
- 后续提升的主要瓶颈，已经更多依赖新的客户样本和更强的 VLM 运行环境，而不是基础架构缺失

## 9. 还有哪些事没有“彻底完成”

这个项目现在是“适合收尾”，不是“完全没有后续工作”。

当前已知限制包括：

- 还缺更多真实客户图纸做外推验证
- `pipeline_isometric` 的 weld list 仍然值得继续做更强的单元格级解析
- 低清晰度双图页仍然主要依赖 review-first / manual 路径
- 当前本地小模型在架构上已经接进来了，但性能仍然受限于开发机

## 10. 如果未来继续做，最顺的续接路径

当后面拿到更多样本或更强硬件之后，最自然的继续方式是：

1. 继续扩充客户真实样本和标注
2. 保持现有 benchmark，不断扩展 truth set
3. 继续硬化 weld list 解析
4. 继续做 BOM 的领域归一化
5. 用同样的 bounded M5 架构接入更强的 24GB 级本地模型或云 VLM adapter

这样做不是推倒重来，而是在现在这套已经跑通的架构上继续增量增强。
